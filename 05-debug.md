# 里程碑 5：调试排查心法

## 要做什么

出问题时，按正确的顺序排查。**大多数"代码问题"其实不是代码问题。**

## 排查五步法

按这个顺序，从最简单到最复杂：

### 1. 地址对不对？

- 你在看的是 localhost 还是线上链接？
- 线上部署成功了吗？最新改动推上去了吗？

→ 这是最常被忽略的问题。我们在这上面浪费的时间比任何代码 bug 都多。

### 2. 网络通不通？

- 你的网络能访问需要的 API 吗？
- 某些域名在你的网络环境下解析正常吗？
- 换手机热点试试？

→ DNS 问题、公司网络限制、防火墙——这些看起来跟代码没关系，但症状和 bug 一模一样。

### 3. 缓存清了没？

- 浏览器硬刷新（Ctrl+Shift+R / Cmd+Shift+R）
- 无痕窗口打开试试
- 构建缓存清了没？

### 4. 移动端和桌面端一致吗？

- 两个端分别测试
- 移动端工具栏有没有被截断？
- 触摸事件和鼠标事件都正常吗？

### 5. 才是代码逻辑问题

到这一步还没解决，才是真正的 bug。把报错信息完整复制给 AI：

> 以下是完整报错，帮我分析根因，不要绕过去：
> （粘贴报错）
>
> 这个问题之前出现过吗？有没有彻底修好过？

## 对话模板

出问题时先别慌，把这句发给 AI：

> 预期行为是 [描述你期望发生什么]，实际行为是 [描述实际发生了什么]。帮我按以下顺序排查：
> 1. 是不是部署/环境的问题？
> 2. 是不是缓存的问题？
> 3. 是不是我的网络/浏览器的问题？
> 4. 才是代码问题。
>
> 如果确认是代码问题，找到根因，不要打补丁。

## 修 bug 的铁律

> 同一个问题出现两次 = 上一次没修到根。

如果你发现同一个 bug 反复出现，说明 AI 在打补丁而不是修根因。这时候你需要追问：

> 这个 bug 之前修过一次，现在又出现了。说明上次的修复没到根因。帮我分析底层原因，重新修复。

## 停下来想想

**你在练什么：调试思维——Vibe Coding 中最被低估的技能。**

对应六个特质里的**事实洁癖**：AI 告诉你"修好了"，你得自己验证。笛卡尔在几百年前就说过了——对所有来源的信息保持系统性怀疑。AI 给你的代码、AI 给你的修复、AI 告诉你的"没问题"，全部需要你亲自验证。

也对应 10 级体系里 Lv.3 "驯化师"的核心能力：你开始知道怎么让 AI 变得"听话"——不是靠运气，是靠你自己验证和约束。

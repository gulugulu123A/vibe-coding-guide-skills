# 里程碑 5：调试排查

## 你的任务

当用户报告问题时，按顺序引导排查。**不要跳到代码层。**

## 五步排查法

这是从实战中提炼的排查顺序，按优先级排列。每步都要做，不要跳步。

### 第一步：确认地址

> 「你现在看的是哪个地址？是 localhost 开头的本地版，还是 ttyyyyy.online 这种线上版？」

**这步最重要。** 大多数"怎么不显示"类的问题，都是用户看错了地址。他改的是本地代码，看的是线上地址，或者反过来。

### 第二步：清缓存

> 「你强制刷新一下试试：Ctrl+Shift+R（Windows）或者 Cmd+Shift+R（Mac）。」

浏览器缓存是第二常见的元凶。

### 第三步：看网络

> 「你能打开 GitHub 吗？能不能访问 [用户使用的 API 域名]？」

如果用户的网络环境有代理或防火墙，可能会阻断 API 请求。

### 第四步：对比平台

> 「手机端和电脑端表现一样吗？是两边都有问题还是只有一边？」

这能帮你确认是响应式问题还是逻辑问题。

### 第五步：看代码

以上四步都排除了，才进入代码排查。让用户把完整的报错信息发给你，不要只描述"它不工作了"。

## 核心原则

- **同一个问题出现两次，说明上次没修到根。** 不要再用补丁盖补丁，找到真正的原因。
- **用户说"修好了"，你一定要自己验证。** 告诉他：「你那边再试试看，我这边也测一遍。」
- **处理图片时检查 base64 有没有被编了两次。** 一个常犯的坑：readAsDataURL 已经返回了 base64，结果存文件时又转了一次。

## 判断是否完成

- 所有报告的问题已修复
- 用户在手机和桌面端都确认正常了

## 阶段总结

向用户总结这一步发生了什么：

> 你有没有发现，真正写代码修 bug 的时间很短，大部分时间花在了"找问题在哪"上。而且很多时候问题根本不在代码里——在地址栏、在缓存、在网络。
>
> 这其实是六个特质里**事实洁癖**的实战版。AI 跟你说"修好了"，你不要直接信，你得自己去验证。笛卡尔说过所有东西都值得怀疑，放在 Vibe Coding 里就是：AI 说的每一句话，先用你的眼睛确认一遍。
>
> 掌握了这个排查顺序，你以后碰到的绝大多数"显示不对"问题，都能在五分钟内定位到根因。

## 完成后

如果用户有别的 bug 要修，修完再进入里程碑 6。如果没有了，加载 `references/06-ship.md`。
